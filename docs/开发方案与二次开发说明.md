# 七维书院活动报名系统开发方案与二次开发说明

> 维护方式：本文是系统后续升级、二次开发、交接和验收的主文档。以后每次新增功能、修复生产问题、调整部署方式，都要同步补充到本文的“升级记录与二开记录”章节，并在相关专题文档中补充细节。

## 1. 文档目标

这份文档解决四件事：

1. 让新开发者快速知道系统由哪些端组成、代码在哪里、数据怎么流转。
2. 让后续升级有固定流程，避免“代码改了但线上没更新”“迁移没跑”“前端静态包没覆盖”等问题。
3. 让二次开发能按统一规范补权限、菜单、接口、数据库迁移、日志和验收。
4. 让运营上线前能按业务闭环验收，而不是只看页面能不能打开。

## 2. 系统定位

本系统是面向书院/活动/课程/共修运营的多端平台，当前包含：

- H5 前台：用户浏览首页、活动、课程、共修、打卡、订单、我的页面。
- 微信小程序：与 H5 共享主要业务逻辑和接口，小程序发布需要单独上传、审核、发布。
- 后台管理端：平台超级管理员、商家管理员、运营、财务、签到人员使用。
- 后端 API：提供登录、权限、商家、活动、课程、订单、支付、装修、打卡、小程序发布等接口。
- 数据库：MySQL/MariaDB，所有结构变更必须通过 TypeORM migration。

当前线上主域名：

- H5：`https://rd.chaimen666.com/`
- 后台：`https://rd.chaimen666.com/admin/`
- API：`https://rd.chaimen666.com/api/`
- 上传文件：`https://rd.chaimen666.com/uploads/`

## 5. 主要业务模块

平台与商家：

- 平台超级管理员管理所有商家、权限、系统设置、上线体检、操作日志、小程序发布。
- 商家管理员只能管理自己商家的活动、课程、共修、装修、订单和会员。
- 所有商家隔离必须同时在接口查询、数据写入、后台筛选和前台 tenantCode 中体现。

活动报名：

- 活动列表、活动详情、票种、优惠码、报名表单、报名审核、签到核销、订单记录。
- 报名链路必须验证：未登录拦截、重复报名、报名状态、订单状态、后台审核和前台状态同步。

课程学习：

- 课程列表、课程详情、课程订单、线下收款确认、我的课程、课时播放、学习记录。
- 未购买课程必须拦截播放；已确认订单后应进入“我的课程”。

共修打卡：

- 共修活动、打卡任务、用户打卡、打卡记录、点赞评论。
- 保存成功必须真正落库；用户刷新后状态要保持。

装修营销：

- 首页装修、页面模块、底部菜单、公告、运营设置。
- 后台装修预览应显示完整前台页面，并突出当前编辑模块。
- 未配置装修时，前台和预览应显示默认装修，不应空白。

会员与用户资产：

- 会员资料、微信身份、openid/unionid、手机号、标签、等级、积分、余额、收藏、评论。
- 小程序用户应保留 AppID/openid/unionid 等平台身份字段，后台会员资料要能看到关键来源信息。

订单财务：

- 活动订单、课程订单、线下收款确认、退款、对账、收款账户、代理结算。
- 未配置真实支付时，不能直接伪造“支付成功”；应进入线下确认或待支付流程。

小程序发布管理：

- 后台保存小程序 AppID、私钥、版本号、版本描述。
- 后端通过 miniprogram-ci 上传体验版、生成体验二维码、提交审核、查询审核状态、发布线上版。
- 仅平台超级管理员或拥有 `miniprogram_release.manage` 权限的账号可操作。
- 每次上传、提审、发布、失败原因都要记录日志。

## 6. 权限设计

权限分两层：

- 后端权限：`apps/api/src/modules/admin/admin-permissions.ts`
- 后台前端权限：`apps/admin/src/permissions.ts`

新增后台功能时，必须同时处理：

1. 在后端权限定义中新增权限 key。
2. 在 `resolveAdminRoutePermission` 中把接口路由映射到权限 key。
3. 在后台 `permissions.ts` 中加入权限组和文案。
4. 在 `router.ts` 中给页面加 `meta.roles` 和 `meta.scope`。
5. 在 `Layout.vue` 菜单中加入入口，并设置 `roles` 和 `scope`。
6. 如果是新增给既有超级管理员的权限，必须加 migration 回填旧账号显式权限列表。
7. 生产部署后要退出重登或强刷，让 `/admin/auth/me` 刷新本地权限缓存。

权限 scope 原则：

- `platform`：仅平台超级管理员可见，例如商家管理、小程序发布、平台系统安全。
- `tenant`：仅商家侧账号可见。
- `tenantOrPlatformAdmin`：商家账号和平台超管都可进入，但数据必须按 tenantId 过滤。
- 平台专属权限不能分配给商家账号。

## 7. 数据库与迁移规范

原则：

- 禁止在生产直接手工改表作为正式方案。
- 所有表结构变更必须新增 migration。
- migration 文件命名使用递增时间戳，例如 `1780920000000-BackfillMiniprogramReleasePermission.ts`。
- migration 必须支持幂等检查，例如 `hasTable`、`hasColumn`，避免重复执行报错。
- 涉及历史数据修复的 migration 要说明影响范围。

常用命令：

```bash
npm --prefix apps/api run migration:show
npm --prefix apps/api run migration:run
npm --prefix apps/api run migration:revert
```

新增实体后，需要确认：

- `apps/api/src/modules/app.module.ts` 已注册实体。
- `apps/api/src/data-source.ts` 已注册实体。
- migration 已创建表、索引和必要字段。
- 后端构建通过。

## 11. Nginx 指向要求

后台 `/admin/` 应指向：

```nginx
location = /admin {
    return 301 /admin/;
}

location ^~ /admin/ {
    alias /www/wwwroot/rd.chaimen666.com/apps/admin/dist/;
    try_files $uri $uri/ /admin/index.html;
}
```

H5 根路径应指向：

```nginx
root /www/wwwroot/rd.chaimen666.com/apps/mobile/dist/build/h5;
```

排查命令：

```bash
nginx -T | grep -A12 -B3 "location ^~ /admin/"
```

## 13. 新增功能开发标准流程

新增后台功能：

1. 明确使用角色：平台、商家、财务、运营、签到。
2. 新增后端 DTO、Service、Controller 接口。
3. 新增实体和 migration，如需要落库。
4. 加权限 key、路由权限映射和操作日志。
5. 新增后台页面、路由、菜单入口。
6. 页面所有危险操作要有确认提示。
7. 接口错误要给用户可理解的中文提示。
8. 构建 API 和后台。
9. 本地走一遍正常流程和异常流程。
10. 提交代码并补本文档的升级记录。

新增 H5/小程序功能：

1. 明确是否必须登录。
2. 明确是否区分 tenantCode。
3. 明确 H5 和小程序是否共用页面。
4. 对未登录、未购买、未报名、未支付、重复提交做状态处理。
5. 保持 2.0 统一头部、底部菜单和视觉风格。
6. H5 构建和小程序构建都要验证。
7. 小程序上线必须重新上传、审核、发布。

新增支付/订单功能：

1. 订单创建必须落库。
2. 未配置支付时不能假成功。
3. 支付回调必须幂等。
4. 金额、状态、用户权益必须一致。
5. 后台财务和用户“我的订单”状态必须同步。
6. 所有收款确认、退款、关闭订单都要写操作日志。

## 14. 验收官验收流程

第一轮：核心业务闭环

- 用户登录。
- 浏览首页装修。
- 活动报名。
- 产生订单。
- 后台审核或确认。
- 用户“我的”状态变化。
- 签到或打卡。
- 后台记录留存。
- 课程下单、确认、进入我的课程、播放和学习记录。

第二轮：权限和多商家隔离

- 超管能看到平台菜单。
- 商家只能看到本商家菜单和数据。
- 无权限菜单不显示。
- 直接访问无权限 URL 应返回 403 或跳转。
- 商家 A 不能看到商家 B 的订单、会员、装修、课程、财务。

第三轮：运营配置

- 后台装修保存后，前台刷新正确变化。
- 默认装修不为空白。
- 底部菜单开关、排序、文案和跳转生效。
- 系统设置、支付设置、报名设置、打卡任务设置保存后真实落库。

第四轮：异常场景

- 未登录访问。
- 重复报名。
- 订单未支付。
- 库存不足。
- 接口失败。
- 刷新页面。
- 切换账号。
- 手机窄屏。
- 权限被取消后再次访问。

第五轮：上线安全和恢复

- `migration:run` 正常。
- API health 正常。
- PM2 online。
- Nginx 配置正确。
- 备份可用。
- 回滚命令可用。
- 日志可查。

## 17. 常见问题排查

后台菜单没有出现：

1. 确认代码是否包含菜单关键字。
2. 确认 `apps/admin/dist/assets/*.js` 是否包含关键字。
3. 确认账号 `/admin/auth/me` 返回权限中是否有对应 key。
4. 确认 Nginx `/admin/` alias 是否指向 `apps/admin/dist/`。
5. 强刷浏览器或退出重登。

构建成功但线上没变化：

1. 确认服务器 HEAD 是否为最新提交。
2. 确认是否构建了正确端，比如后台要构建 `apps/admin`。
3. 确认 Nginx 指向的目录不是旧目录。
4. 确认浏览器缓存是否强刷。

保存成功但刷新丢失：

1. 查接口是否真的写库。
2. 查 GET 接口是否读取同一张表和同一 tenantId。
3. 查前端是否只改了本地状态没调用接口。
4. 查日期、状态、枚举是否被过滤。

支付未配置却显示成功：

1. 查是否走了演示支付分支。
2. 查支付配置是否为空。
3. 查订单状态是否被前端本地改成 paid。
4. 未配置真实支付时，只能进入线下确认或待支付，不能直接成功。

## 18. 小程序发布管理说明

后台入口：

```text
平台端 · 系统安全 -> 小程序发布
```

```text
miniprogram_release.manage
```

需要配置：

- 微信小程序 CI 私钥。
- 版本号。
- 版本描述。

操作流程：

1. 保存小程序发布配置。
2. 构建小程序代码。
3. 上传体验版。
4. 查看体验版二维码。
5. 提交微信审核。
6. 查询审核状态。
7. 审核通过后发布线上版。
8. 后台保留上传、提审、发布、失败原因和操作人记录。

注意：

- H5 更新后构建部署即可生效。
- 小程序必须重新上传、审核、发布后才会对用户生效。
- 不允许做“假提交审核”或“假发布成功”。

## 21. 后续建议路线图

短期必须稳定：

- 完成真实支付接入前，所有订单保持线下确认流程，不做假支付成功。
- 补齐活动报名支付链路的待支付、已确认、已取消、退款状态。
- 补齐课程学习进度、课时记录、已购课程列表闭环。
- 补齐学员动态真实点赞、评论列表和评论审核。
- 完成装修预览和前台渲染一致性验收。

中期运营增强：

- 增加客服查询台，按手机号、订单号、用户、商家快速定位问题。
- 增加财务日报和异常订单提醒。
- 增加商家运营健康分，提示未配置支付、未装修、无课程、无活动等问题。
- 增加小程序发布记录的体验二维码有效期提示和审核失败原因解析。

长期平台化：

- 多商家套餐、权限模板、到期限制。
- 商家独立域名或独立小程序配置。
- 真实微信/支付宝支付和退款闭环。
- 更完整的数据看板和转化漏斗。
- 自动化回归测试覆盖核心用户链路。
